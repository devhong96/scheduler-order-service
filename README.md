## 💳 결제 모듈 설계에 대한 고민
이 서비스의 핵심 기능인 결제 모듈은 **확장성**과 **유지보수성**을 우선으로 고려하여 설계되었습니다. Naver, Kakao, NicePay 등 여러 결제 대행사(PG)를 지원하면서도, 새로운 결제 수단이 추가될 때 기존 코드에 최대한 적게 수정 하는 것에 초점을 두었습니다.

- **추상화를 통한 결합도 감소**: 각기 다른 PG사의 API 명세와 로직을 직접 다루는 대신, 결제 생성, 승인, 취소 등 공통된 행위를 인터페이스로 추상화했습니다. 각 PG사 연동 로직은 이 인터페이스의 구현체로 만들어, 핵심 비즈니스 로직과 결제 연동 코드를 분리했습니다.


- **전략 패턴(Strategy Pattern) 적용**: 사용자가 선택한 결제 수단(Vendor)에 따라 적절한 PG사 구현체를 동적으로 선택하여 결제를 진행하도록 전략 패턴을 적용했습니다. 이를 통해 `OrderService`는 어떤 PG사가 사용되는지 알 필요 없이, 추가된 PG사 관련 코드만 추가하면 됩니다.


- **확장성 확보**: 추후에 새로운 PG사를 추가해야 할 경우, 새로운 구현 클래스를 작성하고 설정 파일에 등록하는 것만으로 간단하게 확장이 가능도록 구조를 설계하였습니다.

이러한 설계를 통해 각기 다른 결제 정책과 API를 가진 여러 PG사를 하나의 일관된 흐름으로 통합하고, 복잡한 결제 로직을 단순하고 명확하게 관리할 수 있게 되었습니다.
